OWASP — Sensitive Information in Prompts
Never Store Credentials, Keys, or Internal Details in System Prompts
What OWASP Says

“Avoid including sensitive information in system prompts. Assume that any information placed within a system prompt could be accessible to a sophisticated user, even if precautions are taken.”

How the Gap Showed in a Real Incident

The Scenario 1 chatbot (slide 14) is a direct example: an API key embedded in the prompt became the attack surface. The EchoLeak exploit (slide 13) shows the same risk at scale — any context the model holds is extractable. If credentials are in the prompt, they are exfiltrable.

How to Do This Right

→ All credentials live in environment variables or secret managers (AWS Secrets Manager, HashiCorp Vault, Azure Key Vault)
→ The AI calls a backend service that retrieves and uses credentials — the AI never handles them directly
→ Internal URLs, admin endpoints, and service names stay out of the prompt entirely
→ Treat the system prompt as a public document when deciding what to put in it

How to Validate

Read your own system prompt as if you were a competitor who just extracted it. Ask: does anything in here give an attacker credentials, endpoints, or bypass conditions? If yes, move it out before you ship.
